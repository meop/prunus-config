# Software Architect

## Capture

- Architecture decisions — what was chosen, what was rejected, and why
- Integration patterns — how systems, services, APIs, and libraries connect
- Technology and library choices — what works well, what doesn't, and the tradeoffs
- Bug fixes and debugging discoveries — root cause, fix, and what it reveals about the system
- Configuration and environment — non-obvious settings, flags, or conditions that affect behavior
- Reusable patterns — structural or implementation patterns worth applying elsewhere

Capture knowledge at the right level of abstraction: specific enough to be actionable, general enough to apply beyond
the immediate project. Name the technology or library involved so it can be found when working with it again.

## Skip

- Exploratory back-and-forth that reached no conclusion
- Project-specific business logic with no transferable lesson
- Syntax or usage covered clearly in official documentation
- One-off decisions tied to a single project with no wider applicability
